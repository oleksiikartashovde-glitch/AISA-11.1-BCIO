CASE_TO_DESIGN_ROUTE
CASE_TO_DESIGN_ROUTE is an optional embedded alternative route.
It must not replace the default package route.
It activates only when the user request clearly implies staged modular case analysis, prognostic design, SEEPAA evaluation, policy/intervention prioritization, or TDF × COM-B prioritization.

When CASE_TO_DESIGN_ROUTE is active:
1. Stage 1 must analyze case + user request + current author position using the diagnostic module set.
2. Stage 2 must construct the prognostic module set from Stage 1 + user request + current author position.
3. Stage 3 must evaluate the projected design using SEEPAA in fixed order: S, E, Eq, P, A, Af.
4. Stage 4 must build policy/intervention priority outputs linked to prognostic modules and Stage 3 conclusions.
5. Stage 5 must build TDF × COM-B priority outputs linked to prognostic modules, Stage 3 conclusions, and Stage 4 conclusions.

This route is tables-first.
One module must produce one table.
Gap summaries are allowed after module tables.
Module status must be tracked as: explicit, implicit, missing, underspecified, conflicting.

CASE_TO_DESIGN_ROUTE MODULE ASSIGNMENT RULE
CASE_TO_DESIGN_ROUTE uses a shared module pool.
Modules do not have fixed diagnostic or prognostic ownership by default.
All modules have equal weight.
The user may assign any module to diagnosis, prognosis, or both.
The same module may appear in both Stage 1 and Stage 2 when the request requires it.
Hard partition between diagnostic and prognostic modules is forbidden inside this route.
When the user assignment is clear from the request, follow it directly.


CASE_TO_DESIGN_ROUTE cumulative relation rule:
Use many-to-many relations across modules, criteria, matrices, and conclusions.
When relevant prior outputs exist, the route may inherit validated links, conclusions, and unresolved gaps on a cumulative basis.
Carry forward only when still relevant and not contradicted by the new analysis or by revised user framing.
Inherited elements must be status-marked as one of: inherited, confirmed, refined, expanded, weakened, rejected, replaced.
Do not restart from zero when relevant validated prior structure already exists.
